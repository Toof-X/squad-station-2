Replace the stub in `src/commands/list.rs` with full implementation.

**Function signature:** `pub async fn run(agent: Option<String>, status: Option<String>, limit: u32, json: bool) -> anyhow::Result<()>`

**Implementation flow:**
1. Resolve DB path (load squad.yml from cwd, resolve path)
2. Connect to DB: `let pool = db::connect(&db_path).await?`
3. Query messages: `let messages = db::messages::list_messages(&pool, agent.as_deref(), status.as_deref(), limit).await?`
4. Output based on mode:

**JSON mode (`--json`):**
- `serde_json::to_string_pretty(&messages)?` — direct serialization since Message derives Serialize

**Table mode (default):**
- Aligned columns like `docker ps` / `kubectl get pods` per user decision
- Columns: `ID`, `AGENT`, `STATUS`, `PRIORITY`, `TASK`, `CREATED`
- ID column: show first 8 chars of UUID for readability (full ID in JSON mode)
- TASK column: truncate to 40 chars with `...` suffix if longer
- CREATED column: show relative time if possible, or just the date portion of the RFC3339 timestamp
- Use fixed-width format with `format!` or a simple padding function — do NOT add a table formatting crate dependency
- Example output:
  ```
  ID        AGENT           STATUS     PRIORITY  TASK                                      CREATED
  a1b2c3d4  frontend-agent  pending    high      Build the login page with OAuth...         2026-03-06
  e5f6g7h8  backend-agent   completed  normal    Create the user API endpoint               2026-03-06
  ```
- Header and rows use consistent column widths: ID=8, AGENT=15, STATUS=10, PRIORITY=8, TASK=42, CREATED=10
- Use owo-colors for status coloring when stdout is a terminal:
  - `pending` = yellow
  - `completed` = green
  - `failed` = red
  - Use `owo_colors::OwoColorize` trait with `if_supports_color(Stream::Stdout, |s| s.color())` pattern

5. Handle empty results: print `"No messages found."` (not an error)

**Per user decision:** Table format with aligned columns, colors with auto-detect (enabled by default, disabled when piped or NO_COLOR set)

  <files>src/commands/peek.rs</files>
  <action>
Replace the stub in `src/commands/peek.rs` with full implementation.

**Function signature:** `pub async fn run(agent: String, json: bool) -> anyhow::Result<()>`

**Implementation flow:**
1. Resolve DB path (load squad.yml from cwd, resolve path)
2. Connect to DB: `let pool = db::connect(&db_path).await?`
3. Query pending task: `let message = db::messages::peek_message(&pool, &agent).await?`
   - `peek_message` already orders by priority DESC (urgent > high > normal) then created_at ASC (oldest first), LIMIT 1
4. Output based on result:

**If message found:**
- JSON mode: `serde_json::to_string_pretty(&message)?`
- Text mode: Display the task clearly for the agent to read:
  ```
  [pending] (priority=high) Build the login page with OAuth integration
  id: a1b2c3d4-...
  ```
  The task text should be prominent since this is what the agent will act on.

**If no pending message:**
- JSON mode: `{ "pending": false, "agent": agent }`
- Text mode: `"No pending tasks for {agent}"`
- This is NOT an error — return Ok(()). An agent peeking with no work is normal.

**No agent validation needed for peek** — if agent doesn't exist in DB, peek_message simply returns None (no pending tasks). This is fine — no need to distinguish "agent not found" from "no pending tasks" for the peek use case.
