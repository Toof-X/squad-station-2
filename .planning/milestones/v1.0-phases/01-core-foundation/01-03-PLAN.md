   - If `json`: `{ "sent": true, "message_id": msg_id, "agent": agent, "priority": priority }`
   - If not json: `"Sent task to {agent} (id={msg_id}, priority={priority})"` — minimal per user decision
     Use owo-colors for a green checkmark if stdout is a terminal: `"✓ Sent task to {agent}"`

**Error cases:**
- Agent not found in DB → clear error, exit 1
- Agent tmux session not running → clear error, exit 1
- tmux send-keys fails → error propagated from tmux module

  <name>Task 2: Implement signal command — idempotent completion update and orchestrator notification</name>
  <files>src/commands/signal.rs</files>
  <action>
Replace the stub in `src/commands/signal.rs` with full implementation.

**Function signature:** `pub async fn run(agent: String, json: bool) -> anyhow::Result<()>`

**Implementation flow:**
1. Resolve DB path (same pattern as send — load squad.yml from cwd)
2. Connect to DB: `let pool = db::connect(&db_path).await?`
3. Validate agent exists: `let agent_record = db::agents::get_agent(&pool, &agent).await?`
   - If None: bail with `"Agent not found: {agent}"`
4. Idempotent status update (MSG-03): `let rows = db::messages::update_status(&pool, &agent).await?`
   - `update_status` already implements `UPDATE ... WHERE status = 'pending' ... LIMIT 1`
   - If rows == 0: no pending message to complete — this is NOT an error (duplicate signal is silently successful)
5. Find orchestrator session name:
   - Query for agent with role='orchestrator': `SELECT name FROM agents WHERE role = 'orchestrator' LIMIT 1`
   - This may need a new helper function `db::agents::get_orchestrator(pool) -> Option<Agent>` — add it to `src/db/agents.rs` if not already present. Use `query_as::<_, Agent>("SELECT * FROM agents WHERE role = 'orchestrator' LIMIT 1").fetch_optional(pool).await?`
6. Notify orchestrator (if rows > 0 — only notify on actual state change):
   - Get the message that was just completed — need the message id for the notification. Option: have `update_status` return the message id along with rows_affected. OR query for the most recently completed message for this agent. Simplest: modify `update_status` in db/messages.rs to return `Option<String>` (the message id) instead of just u64. Use `RETURNING id` clause if sqlx/SQLite supports it, or do a SELECT after the UPDATE.
   - Alternative simpler approach: after the UPDATE, query `SELECT id FROM messages WHERE agent_name = ? AND status = 'completed' ORDER BY updated_at DESC LIMIT 1` to get the task_id for the notification.
   - Build notification: `format!("[SIGNAL] agent={} status=completed task_id={}", agent, task_id)` — structured key=value format per user decision
   - If orchestrator session exists: `tmux::send_keys_literal(&orchestrator_name, &notification)?`
   - If orchestrator session NOT running: signal is persisted in DB, can be retrieved via peek/list on next check (per user decision — "Signals queue in DB regardless of orchestrator availability")
7. Output:
   - If `json`: `{ "signaled": true, "agent": agent, "task_id": task_id_or_null, "orchestrator_notified": bool }`
   - If not json:
     - If rows > 0: `"✓ Signaled completion for {agent} (task_id={task_id})"`
     - If rows == 0: `"✓ Signal acknowledged (no pending task for {agent})"` — silently succeed per MSG-03

**Key design points:**
- Duplicate signals are NOT errors — they silently succeed (MSG-03)
- Orchestrator notification only fires on actual state change (rows > 0)
- If orchestrator session is down, notification is NOT sent but signal is persisted — this is correct behavior per user decision

**DB changes needed:** Add `pub async fn get_orchestrator(pool: &SqlitePool) -> anyhow::Result<Option<Agent>>` to `src/db/agents.rs`. This is a small addition to an existing file from Plan 01.
