# Phase 30 Context: Telegram Integration

**Phase Goal:** When a stall is detected, the user receives a Telegram message on their phone — and a missing or misconfigured Telegram setup never crashes or stalls the watchdog loop

**Created:** 2026-03-24
**Status:** Ready for research/planning

## Prior Decisions (from v2.0 planning + Phase 29)

- Telegram is secondary channel — tmux injection always fires first
- Two separate NudgeState instances (idle + deadlock) — from Phase 29
- DeadlockState debounce (3 cycles default), cooldown (600s default) — from Phase 29
- Watchdog deadlock messages include stuck message IDs, escalation levels — from Phase 29

## Architecture Decision: Delegate Telegram to Orchestrator MCP (Reply-Based)

**Decision:** The Rust watchdog does NOT make HTTP calls to the Telegram Bot API. Instead:

1. Watchdog detects stall → injects message into orchestrator's tmux pane (existing behavior)
2. The injected message includes an **explicit instruction** for the orchestrator to relay via Telegram MCP plugin
3. The orchestrator (Claude Code) handles Telegram delivery using its existing `plugin:telegram` MCP channel
4. No curl, no reqwest, no HTTP client added to the Rust codebase — ever

**Reply-Based Alerting Model (updated 2026-03-24):**
The Telegram MCP plugin's `reply` tool requires a `chat_id` from an **inbound message** — there is no push/broadcast capability. The orchestrator does NOT need to store any chat ID. The flow is:
1. User sends any message to the Telegram bot → orchestrator receives `<channel>` tag with `chat_id`
2. Orchestrator uses that `chat_id` with the `reply` tool to send alerts back
3. If no inbound message has been received (no `chat_id` available), Telegram alerts silently skip — tmux injection remains the primary alert channel
4. This is a **reply-based** flow, not a push-based flow — the user must initiate contact with the bot at least once per session

**Why:** The orchestrator already has a Telegram MCP plugin. Adding an HTTP client to the Rust binary duplicates capability, adds dependencies, and creates a second code path to maintain. The watchdog stays pure (detect + inject), the orchestrator stays responsible for external communication. The reply-based model means zero configuration — no bot tokens, no chat IDs, no env vars in the Rust binary.

**Implication for ALERT-03/ALERT-04:** These requirements MUST be rewritten to reflect delegation architecture. The Rust watchdog is responsible for detection and tmux injection only. The orchestrator client is responsible for Telegram relay.

## Decisions

### 1. Injected Message Format — Explicit Telegram Relay Instruction

**Decision:** Watchdog deadlock messages MUST include an explicit, direct instruction for the orchestrator to relay via Telegram MCP.

**Format per escalation level:**
- Nudge 1: `🚨 [SQUAD WATCHDOG] Deadlock detected — {N} processing message(s) but zero busy agents. Stuck: {msg_ids}. Idle for {duration}m. IMMEDIATELY USE YOUR TELEGRAM MCP PLUGIN TO SEND THIS ALERT TO THE USER. Run: squad-station list --status processing`
- Nudge 2: `🚨 [SQUAD WATCHDOG] Deadlock persists — {N} stuck message(s): {msg_ids}. {duration}m elapsed. IMMEDIATELY USE YOUR TELEGRAM MCP PLUGIN TO ALERT THE USER. Review and re-dispatch or complete manually.`
- Nudge 3 (final): `🚨 [SQUAD WATCHDOG] CRITICAL — deadlock unresolved for {duration}m. Stuck: {msg_ids}. Watchdog stopping alerts. IMMEDIATELY USE YOUR TELEGRAM MCP PLUGIN TO ALERT THE USER. Manual intervention required.`

**Key elements:**
- 🚨 emoji prefix for visual urgency in tmux pane
- Uppercase `IMMEDIATELY USE YOUR TELEGRAM MCP PLUGIN TO SEND THIS ALERT TO THE USER` — direct, unambiguous instruction
- Same actionable content as Phase 29 messages (agent count, message IDs, duration)

**Why:** The orchestrator (Claude Code) reads tmux injections as natural language prompts. An explicit instruction like "USE YOUR TELEGRAM MCP PLUGIN" is the most reliable way to trigger the MCP tool call. Uppercase ensures it's not missed in a long context window.

**Also update:** Prolonged-busy alerts (Pass 3) should also include Telegram relay instruction, as a stuck agent is equally urgent.

### 2. Orchestrator Launch Command — `--channels plugin:telegram`

**Decision:** Update `get_launch_command()` in `init.rs` to include `--channels plugin:telegram` for claude-code orchestrator sessions.

**Implementation:**
- Add `channels: Option<Vec<String>>` field to `AgentConfig` in `config.rs`
- `get_launch_command()` appends `--channels <ch1> --channels <ch2>` for each channel
- Default: orchestrator gets `channels: ["plugin:telegram"]` in squad.yml generated by wizard
- Workers do NOT get channels (they don't need Telegram access)
- squad.yml example:
  ```yaml
  orchestrator:
    provider: claude-code
    model: opus
    channels:
      - "plugin:telegram"
  ```

**Why:** The `--channels` flag is how Claude Code enables MCP plugins at session startup. Without it, the Telegram MCP plugin is not available to the orchestrator, and the relay instruction in watchdog alerts would be ignored.

**Scope:**
- `config.rs`: Add `channels` field to `AgentConfig`
- `init.rs`: Update `get_launch_command()` to append `--channels` flags
- `wizard.rs`: Default orchestrator config includes `plugin:telegram` channel
- `generate_squad_yml()`: Serialize channels list

### 3. Orchestrator Context — Telegram Alert Handling Section

**Decision:** Add a new section to `build_orchestrator_md()` in `context.rs` that instructs the orchestrator how to handle watchdog Telegram relay requests.

**Section content:**
```markdown
## Watchdog Alert Relay — Telegram

When you see a message from `[SQUAD WATCHDOG]` containing `USE YOUR TELEGRAM MCP PLUGIN`,
you MUST immediately use the Telegram MCP `reply` tool to forward the alert to the user.

**How to relay (reply-based model):**
1. Extract the alert content from the watchdog message
2. Use the `chat_id` from the most recent inbound Telegram `<channel>` tag
3. Call the Telegram MCP `reply` tool with that `chat_id` and the alert text
4. Do NOT wait for user response — continue normal operation after sending

**If no inbound Telegram message exists (no chat_id available):**
- Skip Telegram relay silently — the alert is already visible in your tmux pane
- The user can see it via `squad-station ui` or `squad-station browser`
- Do NOT error or halt workflow

**If Telegram MCP is unavailable:**
- Same as above — skip silently, tmux injection is the primary channel
```

**Why:** The orchestrator needs explicit instructions in its context file to know what to do when it sees a watchdog Telegram relay request. The reply-based model means the orchestrator only needs a `chat_id` from an inbound message — no stored configuration, no env vars. If the user hasn't messaged the bot, Telegram alerts are silently skipped.

### 4. Requirement Rewrite — ALERT-03 and ALERT-04

**Decision:** Completely rewrite ALERT-03 and ALERT-04 in REQUIREMENTS.md to reflect the delegation architecture.

**New ALERT-03:** "Watchdog stall alerts injected into the orchestrator's tmux pane include an explicit instruction for the orchestrator to relay the alert to the user via Telegram MCP plugin"

**New ALERT-04:** "The orchestrator's Claude Code session is launched with `--channels plugin:telegram` so the Telegram MCP plugin is available for alert relay. The `channels` field in squad.yml configures which MCP channels are enabled per agent."

**Why:** The original requirements implied the Rust binary would directly call the Telegram Bot API. This is no longer the architecture. Rewriting prevents future agents from mistakenly trying to add HTTP clients back into the Rust codebase.
